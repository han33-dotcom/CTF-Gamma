\chapter*{Executive Summary}
\addcontentsline{toc}{chapter}{Executive Summary}

The Gamma target was fully compromised during the authorized CTF assessment. The compromise began with a misconfigured Network File System (NFS) export that exposed Peter's home directory to the network with read/write access. Because that exported directory contained user-controlled authentication files, write access to the share allowed SSH trust material to be added directly to Peter's account. This produced SSH access as Peter without knowing Peter's password.

After initial access was obtained, multiple independent privilege-escalation paths were identified and validated. Root access was achieved through passwordless \cmd{sudo} access to \cmd{/usr/bin/strace}, unsafe Docker group membership, cron wildcard injection in a privileged backup process, weak user credentials, and a non-root account named \textit{insecurity} with UID 0. These paths show that Gamma was not vulnerable because of one isolated bug. The target suffered from overlapping failures in network service exposure, account management, privilege separation, scheduled task design, and patch management.

An authenticated Nessus scan reinforced the manual findings. The scan reported 251 total findings, including 24 critical and 92 high-severity issues. It also identified the target as Ubuntu 18.04.x, an end-of-standard-support operating system. Although patch debt was significant, the highest-impact attack paths were caused by insecure configuration rather than by a single CVE.

\begin{findingbox}{GammaGold}{Overall Impact}
Gamma should be treated as fully compromised. An attacker on the same network could mount Peter's home directory, gain SSH access, escalate to root, read or modify sensitive files, extract password hashes, create persistence, and recover all CTF flags.
\end{findingbox}

\chapter{Methodology}

The assessment followed a standard attack-chain methodology:
\begin{enumerate}
  \item \textbf{Reconnaissance:} Identify exposed services and open ports.
  \item \textbf{Service analysis:} Determine which services exposed useful attack surface.
  \item \textbf{Vulnerability scanning:} Use authenticated Nessus scanning to identify local weaknesses and patch gaps.
  \item \textbf{Initial access:} Validate whether a network-exposed service could lead to shell access.
  \item \textbf{Privilege escalation:} Enumerate local privileges, users, groups, scheduled tasks, and SUID binaries.
  \item \textbf{Impact validation:} Confirm root access and recover CTF flags.
  \item \textbf{Remediation planning:} Convert validated attack paths into concrete defensive recommendations.
\end{enumerate}

\begin{infobox}{Validation Standard}
Scanner output was treated as supporting evidence, not as final proof. The strongest conclusions in this report are based on manually confirmed attack paths: NFS write access, SSH key injection, local privilege escalation, password compromise, and root-level command execution.
\end{infobox}

\chapter{Initial Access}

\section{NFS Export Discovery}
The first successful access path came from NFS:
\begin{lstlisting}[style=terminal]
showmount -e <Gamma_machine_IP>
\end{lstlisting}

The result showed:
\begin{lstlisting}[style=plaincode]
/home/peter *
\end{lstlisting}

\evidence[0.62\linewidth]{figures/gamma_img-003.png}{NFS export list showing \filepath{/home/peter}.}{fig:nfs-export}

This meant Peter's home directory was exported and accessible to any client matched by \cmd{*}. The wildcard was a critical misconfiguration because it allowed any host on the lab network to mount the share.

\section{Mounting Peter's Home Directory}
The share was mounted from the attacking machine. Once mounted, Peter's files were readable. This exposed user data and one of the CTF flags. The more serious issue, however, was write access: the share allowed files to be created or modified inside Peter's home directory.

\section{SSH Key Injection}
SSH supports public-key authentication through a user's \filepath{authorized_keys} file. If an attacker can write to a user's \filepath{.ssh/authorized_keys}, the attacker can add their own public key and authenticate as that user.

The process was:
\begin{enumerate}
  \item Create a \filepath{.ssh} directory inside Peter's mounted home directory.
  \item Generate or reuse an SSH key pair from the attacking machine.
  \item Place the attacker's public key into Peter's \filepath{authorized_keys} file.
  \item Fix file and directory permissions so SSH accepts the key.
  \item Log in as Peter over SSH.
\end{enumerate}

\evidence{figures/gamma_img-004.png}{Successful SSH session as Peter after adding an authorized key through NFS.}{fig:peter-ssh}

This resulted in SSH access as Peter without knowing Peter's password.

\begin{findingbox}{GammaRed}{Initial Access Finding: Writable NFS Export}

\chapter{Privilege Escalation}

After gaining access as Peter, local enumeration revealed several possible privilege-escalation routes. This section separates successful paths from unsuccessful exploratory paths so the report focuses on confirmed impact.

\section{Validated Path 1: Passwordless sudo strace}
Peter's sudo privileges showed:
\begin{lstlisting}[style=plaincode]
(ALL) NOPASSWD: /usr/bin/strace
\end{lstlisting}

The escalation command was:
\begin{lstlisting}[style=terminal]
sudo strace -o /dev/null /bin/sh
whoami
\end{lstlisting}

\evidence{figures/gamma_img-005.png}{Peter using passwordless \cmd{strace} to spawn a root shell.}{fig:strace}

\cmd{strace} launches and traces another process. If \cmd{strace} is executed as root through \cmd{sudo}, the process it launches also runs with root privileges. By telling \cmd{strace} to execute \cmd{/bin/sh}, Peter effectively launched a root shell.

\begin{findingbox}{GammaRed}{Finding: Passwordless sudo access to strace}
\begin{tabularx}{\linewidth}{L{0.28\linewidth} Y}
\textbf{Severity} & \critical \\
\textbf{Affected user} & Peter \\
\textbf{Result} & Root shell \\
\textbf{Root cause} & Dangerous binary allowed through \cmd{sudo} with \cmd{NOPASSWD} \\
\end{tabularx}
\end{findingbox}

\section{Validated Path 2: Cron Wildcard Injection}
The system contained a root-owned backup workflow that used \cmd{tar} with wildcard expansion over user-controlled files. This is dangerous because shell wildcards are expanded before the target command runs. If filenames begin with \cmd{--}, some tools may interpret them as command-line options rather than ordinary filenames.

The following attacker-controlled filenames were created:
\begin{lstlisting}[style=plaincode]
--checkpoint=1
--checkpoint-action=exec=sh exploit.sh
exploit.sh
\end{lstlisting}

When the backup job executed \cmd{tar} over the directory, the wildcard expanded to include these filenames. \cmd{tar} interpreted them as checkpoint options and executed \filepath{exploit.sh}.

The first payload copied \filepath{/etc/shadow} into Peter's accessible directory. This proved that the cron job was executing attacker-controlled commands as root.

\evidence{figures/gamma_img-006.png}{Cron wildcard injection used to copy shadow data into Peter's directory.}{fig:shadow}

\section{Validated Path 3: Docker Group Abuse}
Peter was a member of the Docker group. Docker group membership is dangerous because the Docker daemon runs with root privileges. A user who can control Docker can often mount the host filesystem or run privileged containers.

\evidence{figures/gamma_img-009.png}{Alpine miniroot filesystem and Docker-related files in Peter's home directory.}{fig:docker-files}

The first attempt to use Docker failed because the expected image was not locally available. The approach was adapted by transferring an Alpine miniroot filesystem into Peter's directory and importing it as a Docker image. Once a usable image was available, Docker could be used to access host resources and achieve root-level control.

\begin{findingbox}{GammaRed}{Finding: Docker group membership}
\begin{tabularx}{\linewidth}{L{0.28\linewidth} Y}
\textbf{Severity} & \critical \\
\textbf{Affected user} & Peter \\
\textbf{Result} & Root-equivalent access \\
\textbf{Root cause} & Docker group membership granted to a non-administrative user \\
\end{tabularx}
\end{findingbox}

\section{Validated Path 4: UID 0 insecurity Account}
During account enumeration, a user named \textit{insecurity} was found with UID 0 and GID 0. In Linux, UID 0 is the root privilege boundary. Any account with UID 0 is root-equivalent even if the username is not \textit{root}.

The password hash for \textit{insecurity} was cracked. The password was:
\begin{lstlisting}[style=plaincode]
P@ssw0rd
\end{lstlisting}

\evidence{figures/gamma_img-010.png}{SSH login as \textit{insecurity}, a UID 0 account.}{fig:insecurity}

The system treated \textit{insecurity} as root because its UID was 0. The account name did not matter. Combined with a weak crackable password, this created a root backdoor.

\begin{findingbox}{GammaRed}{Finding: Non-root account with UID 0}
\begin{tabularx}{\linewidth}{L{0.28\linewidth} Y}

\textbf{Affected account} & \textit{insecurity} \\
\textbf{Password} & \secret{P@ssw0rd} \\
\textbf{Result} & Root shell \\
\textbf{Root cause} & Root-equivalent account misconfiguration \\
\end{tabularx}
\end{findingbox}

\section{Explored but Unsuccessful Paths}
Susan's account was investigated after sensitive information was found in her home directory. Susan used restricted bash (\cmd{rbash}), but editor shell escapes allowed the restricted shell to be bypassed. After escaping the restricted shell, \filepath{/usr/bin/pkexec} was investigated because it was SUID root.

\evidence{figures/gamma_img-012.png}{SUID enumeration while investigating \cmd{pkexec}.}{fig:pkexec}

CVE-2021-4034, known as PwnKit, was considered because it affects vulnerable versions of \cmd{pkexec}. The exploit was downloaded and compiled, but it did not successfully escalate privileges on Gamma. This remained an explored path, not a validated finding.

The target was also running a Linux 4.15.x kernel. Public exploits related to this kernel range were investigated, including an Exploit-DB entry associated with CVE-2018-18955, but the exploit did not succeed.

\evidence{figures/gamma_img-013.png}{Public Linux 4.15.x exploit reference investigated during assessment.}{fig:kernel}

SUID binaries were enumerated, but no writable or directly abusable SUID binary was identified during the initial pass.

\chapter{Full Attack Chain Summary}

\begin{enumerate}
  \item Nmap identified SSH, rpcbind, and NFS.
  \item \cmd{showmount} revealed that \filepath{/home/peter} was exported over NFS.
  \item The NFS share was mounted from the attacking machine.
  \item The share allowed write access to Peter's home directory.
  \item SSH authorized-key material was added to Peter's account.
  \item SSH access as Peter was obtained without Peter's password.
  \item Local enumeration identified dangerous privilege-escalation paths.
  \item Root was obtained through multiple independent methods, including \cmd{sudo strace}, cron wildcard injection, Docker group abuse, Bob's sudo access, and the UID 0 \textit{insecurity} account.
  \item All CTF flags were recovered.
\end{enumerate}

\begin{infobox}{Attack Chain Interpretation}
The NFS exposure was the initial failure, but the full compromise was enabled by several additional local misconfigurations. Removing only one weakness would not be enough to harden the system; the remediation effort must address network exposure, account policy, local privilege rules, scheduled jobs, and patch management together.
\end{infobox}

\chapter{Findings and Risk Analysis}

\section{Finding 1: Writable NFS Export Enabled Account Takeover}
\begin{findingbox}{GammaRed}{Critical Finding}
\begin{tabularx}{\linewidth}{L{0.24\linewidth} Y}
\textbf{Component} & NFS \\
\textbf{Evidence} & \filepath{/home/peter *} in NFS export list \\
\textbf{Exploitation} & Mounted share, wrote SSH key material, logged in as Peter \\
\textbf{Impact} & Unauthorized SSH access as Peter \\
\end{tabularx}
\end{findingbox}

The target exported Peter's home directory over NFS using wildcard client access. The share was writable, allowing files to be created and modified inside Peter's account directory. Because SSH trusts the \filepath{authorized_keys} file in a user's home directory, this allowed direct account takeover.

\textbf{Recommendation:} Remove the export if it is not required, restrict NFS clients to explicit trusted hosts, avoid exporting user home directories, use read-only exports where possible, and audit Peter's \filepath{.ssh} directory.

\section{Finding 2: Passwordless sudo strace Allowed Root Shell}
\begin{findingbox}{GammaRed}{Critical Finding}
\begin{tabularx}{\linewidth}{L{0.24\linewidth} Y}
\textbf{Component} & sudo \\
\textbf{Evidence} & \cmd{(ALL) NOPASSWD: /usr/bin/strace} \\
\textbf{Exploitation} & \cmd{sudo strace -o /dev/null /bin/sh} \\
\textbf{Impact} & Immediate root shell \\
\end{tabularx}
\end{findingbox}

Peter could run \cmd{strace} as root without a password. Since \cmd{strace} can launch another program, this permission allowed Peter to launch \cmd{/bin/sh} as root.

\textbf{Recommendation:} Remove the \cmd{NOPASSWD} rule and avoid granting sudo access to debuggers, interpreters, editors, pagers, archive utilities, or tools with shell escape functionality.

\section{Finding 3: Cron Wildcard Injection Enabled Root Command Execution}
\begin{findingbox}{GammaRed}{Critical Finding}
\begin{tabularx}{\linewidth}{L{0.24\linewidth} Y}
\textbf{Component} & Cron / backup script \\
\textbf{Evidence} & Root-owned backup used \cmd{tar} wildcard expansion \\
\textbf{Exploitation} & Injected \cmd{--checkpoint-action=exec=sh exploit.sh} \\
\textbf{Impact} & Arbitrary command execution as root \\
\end{tabularx}
\end{findingbox}

A root-owned scheduled backup process used wildcard expansion over a directory writable by Peter. Malicious filenames were interpreted as \cmd{tar} options, causing the backup job to execute attacker-controlled commands.

\textbf{Recommendation:} Avoid wildcard expansion in root-owned scripts, do not run backup jobs as root unless required, use explicit file lists or safely quoted paths, and ensure user-writable directories are never consumed unsafely by privileged scripts.

\section{Finding 4: Docker Group Membership Granted Root-Equivalent Access}
\begin{findingbox}{GammaRed}{Critical Finding}
\begin{tabularx}{\linewidth}{L{0.24\linewidth} Y}
\textbf{Component} & Docker \\
\textbf{Evidence} & Peter was in the Docker group \\
\textbf{Exploitation} & Imported local Alpine filesystem and used Docker for host access \\
\textbf{Impact} & Root-equivalent control \\
\end{tabularx}
\end{findingbox}

Peter's Docker group membership allowed interaction with the Docker daemon. Because the daemon runs with root privileges, Docker access can often be used to mount or manipulate host filesystems.

\textbf{Recommendation:} Remove non-administrative users from the Docker group, use rootless Docker where possible, monitor Docker socket access, and treat Docker group membership as privileged access.

\section{Finding 5: UID 0 insecurity Account Created Root Backdoor}
\begin{findingbox}{GammaRed}{Critical Finding}
\begin{tabularx}{\linewidth}{L{0.24\linewidth} Y}
\textbf{Component} & Local accounts \\
\textbf{Evidence} & \textit{insecurity} account with UID 0 \\
\textbf{Password} & \secret{P@ssw0rd} \\
\textbf{Impact} & Direct root-equivalent login \\
\end{tabularx}
\end{findingbox}

The system contained a non-root account with UID 0. Linux grants root privileges based on UID, not username. Therefore, \textit{insecurity} was functionally equivalent to root.

\textbf{Recommendation:} Remove the \textit{insecurity} account or change it to a non-root UID. Ensure only \textit{root} has UID 0.

\section{Finding 6: Weak User Passwords Enabled Account Compromise}
\begin{findingbox}{GammaRed}{Critical Finding}
\begin{tabularx}{\linewidth}{L{0.24\linewidth} Y}
\textbf{Component} & SSH / local accounts \\
\textbf{Evidence} & Bob's password found with Hydra \\
\textbf{Password} & \secret{secret} \\
\textbf{Impact} & SSH access as Bob, then root via sudo \\
\end{tabularx}
\end{findingbox}

Bob used a weak password that was found through a wordlist attack. Bob also had unrestricted sudo privileges, which made the password compromise a direct path to root.

\textbf{Recommendation:} Enforce strong password policy, enable lockout or throttling, disable SSH password authentication if possible, and review sudo access for all users.

\section{Finding 7: Unsupported Ubuntu Baseline and Patch Debt}
\begin{findingbox}{GammaOrange}{High Finding}
\begin{tabularx}{\linewidth}{L{0.24\linewidth} Y}
\textbf{Component} & Operating system and packages \\
\textbf{Evidence} & Nessus reported Ubuntu 18.04.x SEoL and many critical/high findings \\
\textbf{Impact} & Increased exposure to known vulnerabilities \\
\end{tabularx}
\end{findingbox}

The target ran an old Ubuntu baseline with significant package vulnerabilities. While not every CVE was exploited, the patch state increased the number of viable attack paths.

\textbf{Recommendation:} Upgrade to a supported Ubuntu release, apply kernel and package security updates, remove unused services and packages, and re-run Nessus after patching.

\section{Finding 8: SSH Password Authentication Increased Remote Attack Surface}
\begin{findingbox}{GammaOrange}{High Finding}
\begin{tabularx}{\linewidth}{L{0.24\linewidth} Y}
\textbf{Component} & SSH \\
\textbf{Evidence} & SSH accepted password authentication \\
\textbf{Impact} & Password guessing and credential reuse risk \\
\end{tabularx}
\end{findingbox}

SSH accepted password authentication. This increased the effectiveness of weak credentials, as demonstrated through Bob's account.

\textbf{Recommendation:} Disable password authentication if operationally feasible, require SSH keys, restrict SSH access by IP or subnet, and enable alerting for failed login attempts.

\section{NFS Hardening}
\begin{itemize}
  \item Do not export user home directories unless there is a controlled operational requirement.
  \item Replace wildcard client access with explicit trusted hosts or subnets.
  \item Prefer read-only exports when write access is not required.
  \item Use \cmd{root_squash}; verify that \cmd{no_root_squash} is not used.
  \item Audit exported directories for SSH keys, scripts, secrets, and writable startup files.
\end{itemize}

\section{Privilege and Account Hardening}
\begin{itemize}
  \item Remove passwordless sudo permissions unless absolutely required.
  \item Do not grant sudo access to binaries with shell escapes or arbitrary command execution.
  \item Review \filepath{/etc/sudoers} and files under \filepath{/etc/sudoers.d/}.
  \item Confirm that only \textit{root} has UID 0.
  \item Reset weak passwords and rotate compromised SSH keys.
\end{itemize}

\section{Docker Hardening}
\begin{itemize}
  \item Treat Docker group membership as administrative access.
  \item Remove normal users from the Docker group.
  \item Use rootless Docker where container access is required without host-level privilege.
  \item Monitor Docker socket access and privileged container creation.
\end{itemize}

\section{Cron Hardening}
\begin{itemize}
  \item Avoid shell wildcards in root-owned scripts that operate on user-controlled directories.
  \item Use explicit file lists or safely quoted paths.
  \item Run scheduled jobs under dedicated low-privilege service accounts.
  \item Restrict write access to directories consumed by scheduled jobs.
\end{itemize}

\section{SSH and Patch Management}
\begin{itemize}
  \item Disable SSH password authentication if key-based authentication is feasible.
  \item Deny direct root SSH login.
  \item Enforce strong password policy and account lockout.
  \item Upgrade the system to a supported Ubuntu release.
  \item Apply kernel and package security updates.
  \item Re-run authenticated vulnerability scans after patching.
\end{itemize}
